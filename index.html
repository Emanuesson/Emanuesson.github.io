LibGDV

Description

Working in a scientific environment requires a fast, open and simple approach in software-design that is flexible enough to get extended to a full scale project in the end. The Gtk+ library offers many of these features in terms of UI-programming and the underlying GLib and Gobject-framework is a powerful approach to use object-oriented programming (OOP) mostly independent of the available platform or programming language. Despite the advantages, most free scientific codes do not apply Gtk+ even in applications were the integration would be highly preferable. Gdv tries to provide a part of the functionality to make Gtk+ a little more interesting for people that are looking for a way to observe and control technical environments.

I startet to work on gdv because I was pretty annoyed by the options that are available for the visualisation of scientific and technical data, such as forcing Gnuplot within a GtkWidget-instance (this approach limited to systems with a working X11-platform), or using really old libraries such as GtkExtra that are now so outdated that they require functions deprecated 17 years ago, or a rather specialized library such as GOffice that re-implements a lot of the overhead that has been removed or improved in Gtk+ over the years, leading to rather awkward solutions for things one would expect as elementary functionality of any graphic library based on Gtk+, such as theme-management or geometry-handling.

Current functionality

For now the functions provided by Gdv are still really limited. However a good starting point for applications is provided by the documentation

Gevse

Gevse is a Gnome-based EVSE-system (electric vehicle charging station). It was influenced by similar projects, such as the SmartEVSE. The general aim is the load management and the logging of power consumption for different vehicle charging stations on a single raspberry pi. A lot of the planned functionality is still missing. For the moment it's actual nothing more than a nice little electricity meter with an underlying sqlite-database and some functionality for system maintenance based on Gtk+ and broadway.

The more general aim

When I first had my hands on a raspberry pi and installed ARM-Arch-Linux on it, I was fascinated about the functionality and possibility of this small device. I was even more excited when I realized that the performance of an raspberry pi was absolutely sufficient to use it as a small broadway-server and start normal Gtk+-applications on it.

Broadway is nevertheless now something like a lost child in the development of Gtk+ and misses major functionality and has definitely safety issues that prevent a general application in a serious environment. But the general idea to use desktop-applications from within your browser for control and interaction of a distributed system is still an highly recent development. I was hoping to build first of all an Gtk+-application that allowed me to manage a system from a single window and then use this application from within the browser and extend it via a simple plugin for the physical purpose of this one device. E.g. this device might be a raspberry pi, connected to your local network, sitting on a wall and managing different charging stations for electric vehicles... Or it might be a set of sensors in a vehicle, a building or an industrial process... etc.

Building an application that offers the possibility to safely manage and maintain a whole system in an integral way is a challenging task alone. I initially hoped, I could use some of the existing programs of gnome, such as Gnome-Terminal, Nauitlus, Gedit, etc., to build such an application around these apps. Gtk+ offers for this purpose the GtkSocket and GtkPlug-widgets that are limited to X11. And that is the problem. These connectors are naturally not available from within a broadway-session. Beside this the development of Gtk+ is sometimes rapidly changing and has seen some new directions in the last couple of years. One of these new directions is the jump from X11 to wayland. With this new system and some new emerged broadway alternatives such as westfield it might be a good possibility to finally build a Gtk+-based distributed system maintenance-software that could be safely implemented in a web-framework such as Node.js and relies on standard gnome-apps.

User interface

For now this is just a really small Gtk+-app with a lot of placeholders for the different functionalities. The whole thing uses broadway as backend. The current focus should be to change this particular issue and play around for a while with e.g. westfield, as described above.

Data management

For the moment data logging is just implemented for simple S0-interfaces. This functionality can be expended in the future to more complex interfaces and it should be organized within the logging-scripts.

User management

Users apply to the charging process via phone-call, a user interface, automatically via the license plate or by an human administrator. However this functionality is at the moment still completely missing. And due to the different techniques that are necessary for this achievement there is no focus on this part.
